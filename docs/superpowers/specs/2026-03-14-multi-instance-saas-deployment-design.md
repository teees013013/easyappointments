# Multi-Instance EasyAppointments for SaaS Deployment

**Date:** 2026-03-14
**Status:** Approved
**Approach:** Strategy B (Multiple Instances, Cleanest Isolation)

## Problem

EasyAppointments is single-tenant. We need to use it as shared booking infrastructure across 4-10 SaaS products, each with full data isolation, separate branding, separate API tokens, and separate webhook configurations.

## Solution

Build a single production Docker image from this fork. Deploy each product as a separate Coolify service using the same image with per-instance environment variables. Share one MySQL server with separate databases per product.

## Architecture

```
                        Coolify Server (Traefik Reverse Proxy)
                        ├── product-a.bookings.yourdomain.com
                        │   └── EA Container A (env: EA_DB_NAME=ea_product_a)
                        ├── product-b.bookings.yourdomain.com
                        │   └── EA Container B (env: EA_DB_NAME=ea_product_b)
                        ├── product-c.bookings.yourdomain.com
                        │   └── EA Container C (env: EA_DB_NAME=ea_product_c)
                        └── Shared MySQL 8.0 Container
                            ├── Database: ea_product_a
                            ├── Database: ea_product_b
                            └── Database: ea_product_c
```

Each container runs Nginx + PHP-FPM via supervisord, exposing port 80. Coolify's Traefik handles SSL termination and subdomain routing.

## Integration Points Per Instance

| Integration | How It Works |
|---|---|
| REST API (`/api/v1/`) | Product backend calls with Bearer token (unique per instance) |
| Booking page embed | iFrame embed at `https://<subdomain>/booking` |
| Webhooks | Register per-product webhook URLs with unique secret tokens |
| Google Calendar sync | Per-instance Google OAuth credentials via env vars |
| CalDAV sync | Per-provider CalDAV configuration via admin UI |
| CORS | Restricted to product domains via `EA_CORS_ORIGINS` env var |

## Environment Variables

Each Coolify service is configured with these env vars:

| Variable | Required | Default | Purpose |
|---|---|---|---|
| `EA_BASE_URL` | Yes | `http://localhost` | Instance's public URL |
| `EA_DB_HOST` | Yes | `mysql` | MySQL hostname (Coolify internal network) |
| `EA_DB_NAME` | Yes | `easyappointments` | Database name for this instance |
| `EA_DB_USERNAME` | Yes | `user` | MySQL username |
| `EA_DB_PASSWORD` | Yes | `password` | MySQL password |
| `EA_CORS_ORIGINS` | No | `*` (wildcard) | Comma-separated allowed CORS origins. **Should be set per instance in production.** Falls back to wildcard for dev/backwards compatibility. |
| `EA_FRAME_ANCESTORS` | No | `*` | CSP frame-ancestors for iframe embedding |
| `EA_GOOGLE_SYNC` | No | `false` | Enable Google Calendar sync |
| `EA_GOOGLE_CLIENT_ID` | No | (empty) | Google OAuth client ID |
| `EA_GOOGLE_CLIENT_SECRET` | No | (empty) | Google OAuth client secret |
| `EA_LANGUAGE` | No | `english` | Default language |
| `EA_DEBUG_MODE` | No | `false` | Enable debug mode |
| `EA_ENCRYPTION_KEY` | Yes | (none) | Unique encryption key per instance (used for session encryption). Generated by provisioning script. |
| `EA_MAIL_PROTOCOL` | No | `smtp` | Email protocol (mail, smtp, sendmail) |
| `EA_SMTP_HOST` | No | (empty) | SMTP server hostname |
| `EA_SMTP_PORT` | No | `587` | SMTP server port |
| `EA_SMTP_USER` | No | (empty) | SMTP username |
| `EA_SMTP_PASS` | No | (empty) | SMTP password |
| `EA_SMTP_CRYPTO` | No | `tls` | SMTP encryption (ssl, tls) |
| `EA_MAIL_FROM_ADDRESS` | No | (empty) | From address for outbound emails |
| `EA_MAIL_FROM_NAME` | No | (empty) | From name for outbound emails |
| `EA_PROXY_IPS` | No | (empty) | Trusted proxy IPs for Traefik (e.g., `172.16.0.0/12`) |

## File Changes

### New Files

#### `Dockerfile.production`

Multi-stage production build:

**Stage 1 (builder):**
- Base: `php:8.4.3-fpm` (pinned version)
- Installs Composer and npm
- Runs `composer install --no-dev --optimize-autoloader`
- Runs `npm install && npx gulp compile`
- Produces `vendor/` and `assets/vendor/`

**Stage 2 (production):**
- Base: `php:8.4.3-fpm-alpine3.21` (pinned version)
- Installs Nginx and supervisord
- Installs runtime PHP extensions only: `gd intl ldap mbstring mysqli pdo_mysql soap xml zip bcmath gettext`
- Copies application source, vendor/, and built assets from stage 1
- Creates storage directories with proper permissions:
  ```dockerfile
  RUN mkdir -p /var/www/html/storage/{logs,cache,sessions,uploads,backups} \
      && chown -R www-data:www-data /var/www/html/storage
  ```
- Entrypoint script runs `php index.php console migrate` before starting supervisord (handles automatic DB migrations on deployment)
- Runs supervisord (manages Nginx on port 80 + PHP-FPM on port 9000)
- Exposes port 80
- Health check: `HEALTHCHECK --interval=30s --timeout=5s CMD curl -f http://localhost/booking || exit 1`

**Note:** `storage/uploads` and `storage/backups` should be configured as Docker volumes in Coolify for data persistence across container restarts.

#### `docker/production/nginx.conf`

Production Nginx configuration:
- Listens on port 80
- Serves static assets with cache headers
- Proxies PHP to PHP-FPM on 127.0.0.1:9000
- `client_max_body_size 20M` (production-appropriate)
- Explicit deny rules:
  ```nginx
  location ~ /(config\.php|config-sample\.php|composer\.(json|lock)) {
      deny all;
  }
  location ~ ^/storage/ {
      deny all;
  }
  location ~ /\.ht {
      deny all;
  }
  ```

#### `docker/production/supervisord.conf`

Supervisord configuration running:
- `nginx` (foreground)
- `php-fpm` (foreground)

#### `docker/production/entrypoint.sh`

Container entrypoint script that:
1. Sets correct permissions on `storage/` directories
2. Runs `php index.php console migrate` (applies any pending DB migrations)
3. Starts supervisord

This ensures migrations run automatically on every deployment/restart.

#### `scripts/provision-instance.sh`

Shell script that:
1. Accepts product name as argument
2. Checks if database already exists (idempotent — skips creation if exists)
3. Connects to MySQL server
4. Creates database `ea_<product_name>` (if not exists)
5. Creates a dedicated MySQL user for the instance (if not exists)
6. Grants all privileges on that database
7. Generates a random `EA_ENCRYPTION_KEY`
8. Outputs the complete set of env vars to configure in Coolify

Usage: `./scripts/provision-instance.sh product-d`

Script is idempotent — safe to re-run without errors.

#### `.dockerignore`

Excludes from production image (specific dev directories, not blanket `docker/`):
- `docker/php-fpm/` (dev PHP-FPM config)
- `docker/nginx/` (dev Nginx config)
- `docker/mysql/` (dev MySQL data)
- `docker/baikal/` (dev CalDAV server)
- `docker/openldap/` (dev LDAP server)
- `docker-compose.yml`
- `tests/`
- `node_modules/`
- `.git/`
- `docs/`
- `storage/logs/*`
- `storage/sessions/*`
- `storage/cache/*`
- `*.md`
- `phpunit.xml`

**Note:** `docker/production/` is NOT excluded — it contains the production nginx, supervisord, and entrypoint configs needed by `Dockerfile.production`.

#### `application/config/routes.php`

Replace permissive CORS with env-var-driven allowlist and fix security issues:

```php
// CORS: Require explicit origin configuration (no default wildcard)
$allowed_origins_raw = getenv('EA_CORS_ORIGINS') ?: '';
$origin = $_SERVER['HTTP_ORIGIN'] ?? '';

if (!empty($allowed_origins_raw)) {
    $allowed_origins = array_filter(array_map('trim', explode(',', $allowed_origins_raw)));

    if (in_array($origin, $allowed_origins, true)) {
        header('Access-Control-Allow-Origin: ' . $origin);
        header('Access-Control-Allow-Credentials: true');
    }
} else {
    // No CORS origins configured — allow all (dev/backwards compat)
    header('Access-Control-Allow-Origin: *');
    // Note: Do NOT set Allow-Credentials with wildcard origin (browsers reject this)
}

if (isset($_SERVER['HTTP_ACCESS_CONTROL_REQUEST_METHOD'])) {
    header('Access-Control-Allow-Methods: GET, POST, PUT, PATCH, DELETE, OPTIONS, HEAD');
}

if (isset($_SERVER['HTTP_ACCESS_CONTROL_REQUEST_HEADERS'])) {
    // Explicit allowlist instead of reflecting arbitrary headers
    header('Access-Control-Allow-Headers: Content-Type, Authorization, X-Requested-With');
}

if (isset($_SERVER['REQUEST_METHOD']) && $_SERVER['REQUEST_METHOD'] === 'OPTIONS') {
    exit(0);
}
```

Add Content-Security-Policy frame-ancestors header:

```php
$frame_ancestors = getenv('EA_FRAME_ANCESTORS') ?: '*';
if ($frame_ancestors !== '*') {
    header("Content-Security-Policy: frame-ancestors 'self' " . $frame_ancestors);
}
```

## Provisioning Workflow

### Adding a New Product Instance

1. **Run provisioning script:**
   ```bash
   ./scripts/provision-instance.sh my-product
   ```
   Output: database created, credentials and env vars generated (including random `EA_ENCRYPTION_KEY`).

2. **Deploy in Coolify:**
   - Create new service from Docker image
   - Set subdomain: `my-product.bookings.yourdomain.com`
   - Paste env vars from provisioning script output
   - Configure `storage/uploads` and `storage/backups` as Docker volumes
   - Deploy

3. **Configure EasyAppointments:**
   - Visit `https://my-product.bookings.yourdomain.com/installation`
   - Set admin credentials, company name, branding
   - Configure services and providers
   - Generate API token (Settings > API Token)
   - Register webhooks for product backend

### Upgrading All Instances

1. Pull upstream changes into fork
2. Rebuild Docker image: `docker build -f Dockerfile.production -t ea-production:latest .`
3. Push to container registry
4. Redeploy each Coolify service — the entrypoint script runs `php index.php console migrate` automatically on startup, applying any pending migrations

## Security

| Concern | Mitigation |
|---|---|
| CORS | `EA_CORS_ORIGINS` required per instance. No default wildcard. `Access-Control-Allow-Credentials` only sent with explicit origin match. |
| CORS headers | `Access-Control-Allow-Headers` uses explicit allowlist, not reflected from request. |
| iframe embedding | `EA_FRAME_ANCESTORS` + CSP header |
| API auth | Unique Bearer token per instance (set in admin UI) |
| Webhook auth | Per-webhook secret header + token |
| DB isolation | Separate database per instance, dedicated MySQL user per instance |
| Encryption key | Unique `EA_ENCRYPTION_KEY` per instance (generated by provisioning script). Prevents cross-instance session forgery. |
| Session cookies | Each instance runs on its own subdomain, preventing cookie collision across instances. |
| SSL | Handled by Coolify/Traefik with auto Let's Encrypt |
| Sensitive files | Nginx denies access to `config.php`, `config-sample.php`, `composer.json`, `composer.lock`, `storage/` directory |
| Reverse proxy | `EA_PROXY_IPS` configured to trust Traefik's Docker network range |

## Known Limitations and Notes

- Each instance requires its own admin setup (no centralized management)
- Upgrading requires redeploying N services (one per product)
- No shared customer pool across instances (by design — this is the isolation you want)
- Google Calendar OAuth requires separate credentials per instance (one Google Cloud project can issue multiple OAuth client IDs)
- LDAP authentication is supported but not covered by env vars in this design. If needed, add `EA_LDAP_*` env vars and modify `application/config/constants.php` and the LDAP controller.
